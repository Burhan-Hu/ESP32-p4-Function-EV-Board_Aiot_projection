智学 Guardian 方案讨论稿1 项目概述
1.1 选题定位
本方案将原“通用流视频多模态 AIoT 智能体终端”的宏观能力，收窄到一个更具应用价值和比赛可落地性的场景：
学生网课与家庭作业。系统面向家长不在身边、学生独立上网课或完成作业的学习环境，通过 ESP32-P4 采集桌面视频、
环境传感器和课程音频，在本地完成低成本学习状态初筛，在云端完成课程内容总结和复杂语义理解，并通过本地屏幕、
语音、蜂鸣器、灯光或家长端报告形成反馈闭环。
本项目的定位不是“儿童监控设备”，也不是“心理状态诊断系统”。它更适合定义为“学习陪伴型 AIoT 终端”：
系统只根据外部可观察线索识别疑似分心、离座、切屏、长时间无书写等学习状态，不对学生的心理状态、学习能力或疾
病作判断。这样的定位更符合校园 /家庭教育场景的伦理边界，也更容易在比赛中被接受。
1.2 背景与痛点
在线教育、网课复习和家庭自习已 经成为学生学习的重要形 式。家长或 老师不在现场时，学生是 否持续在课程 页面、
是否离座、是 否玩手机、是 否长时间 趴桌、是 否真正完成作业， 往往只能 依赖事后询问。传统摄像头可以录像， 但缺乏学
习行为理解和课程内容总结能力 ；普通学习 App 可以记录课程 播放时长， 却无法感知 真实桌面状态 ；语音 助手可以 回答问
题，却无法长 期观察学习场景。
因此，本项目 希望构建一套“端侧 常驻感知、云端语义理解、本地物理联 动”的学习场景 AIoT 系统。端侧 ESP32-
P4 不持续上传完 整视频，而是在本地完成 画面变化、人体存在、屏幕状态、作业 区域变化等轻量判断 ；只有当用户主动
提问、课程结 束或疑似分心事件 触发时， 才上传关键帧、课程音频 片段和结构 化事件摘要， 由云端智能体生成课程总结和
学习建议。
1.3 目标用户与实验边界
项目 说明

性反馈的家长
第一版场景桌面 左前方部 署一个 ESP32-P4 终端，摄像头 覆盖学生上 半身、

1.4 项目创新点
第一，场景 聚焦：把宏观的“通用 空间智能体”落到家庭网课和作业桌面， 问题清晰、演示 直观、用 户价值 明确。
第二，端云 协同：ESP32-P4 负责高频轻量任务，云端 VLM/LLM 只处理低频复杂理解， 避免把系统 做成单纯的“ 大
模型 API 套壳”。
第三，课程总结智能体：系统不 仅识别学习状态， 还能对课程音频、屏幕关键帧和事件记录 进行整理， 输出学生/家
长都能读懂的学习报告。
第四，隐私友好： 默认保存结构 化事件、关键帧和 短时缓存，家长端 优先展示学习摘要而不是 连续视频， 减少对学生
的压迫感。
2 应用场景与需求分析
2.1 家庭网课场景

智学 Guardian 方案讨论稿场景设定为学生在家中 使用电脑或平 板上网课，家长 因工作无法 全程陪 同。系统部 署在显示器上方或桌面 左前方，摄

2.2 作业自习场景
学生完成 纸质作业或课 后练习时，系统重点关注桌面作业 区域。通过关键帧和 区域变化检测，系统判断是 否持续有书
写行为、作业本是 否长时间 空白、是 否频繁摆弄无关物 品。对 于轻微分心，系统 优先给学生本 人温和提醒 ；对于长时间离
座或 高频无关行为， 才生成事件摘要 发送给家长端。
2.3 家长反馈需求
家长 真正需要的不是实时 盯着孩子的摄像头 画面，而是课 后能够知道三个问题：这 节课大概讲了什么，孩子 有没有明
显离开或分心， 后面该关注 哪些学习点。 因此系统的家长端不以 直播为核心，而以“课程总结 + 学习状态时间线 + 建议提
醒”为核心。

3 系统总体架构
系统采用“多模态感知层 - ESP32-P4 边缘处理层 - 云端学习智能体层 - 物联网反馈层”的 四层架构。多模态感知层 负
责采集摄像头、麦克风、 人体存在、光 照、压力/在座等 数据；ESP32-P4 负责采集、 缓存、事件初筛、关键帧 选择和本地
提醒 ；云端学习智能体 负责视觉问答、课程音频 转写、课程总结和家长报告生成 ；物联网反馈层 负责屏幕 显示、语音 播
报、蜂鸣器、RGB 灯、 继电器和 手机端通知。

5 核心功能设计
5.1 F1 入座与离座检测
系统通过视 觉人体检测、PIR 人体红外和可 选座椅压力传感器判断学生是 否处于学习位 置。若检测到学生离座 超过设
定阈值， 例如 3 分钟，系统记录离座事件 ；若课程 仍在进行， 则本地屏幕提示“ 请回到课程”。 该功能 强调“在不在学习
位”这一 客观状态，不 直接推断学生的学习态 度。
5.2 F2 屏幕课程状态检测
摄像头并不需要完 整读取屏幕内容，只需要在关键帧中判断屏幕是 否处于课程 相关页面。 第一版可通过云端 
VLM/OCR 判断 页面是 否为网课 播放器、课件、 文档或无关 娱乐页面；端侧 负责选取清晰关键帧。 若连续多次识别为无关
页面，系统 先本地温和提醒， 再记录事件。
5.3 F3 疑似分心行为识别
系统识别的是外部可观察行为线索，而不是心理状态。 典型规则包括：连续 60 秒未面向屏幕 且无书写 动作，判定为
疑似分心 ；低头 看手机超过 30 秒，判定为 手机干扰；趴桌超过 45 秒，判定为 疲劳风险 ；桌面无书写 动作超过 5 分钟，判
定为作业 停滞。
5.4 F4 课程音频采集与课程总结
课程 进行时，ESP32-P4 按时间 片缓存音频。课程结 束或用 户触发后，系统将音频 片段转写为 文本，并结合屏幕关键
帧中的课件 标题、板书或 公式，生成本 节课知识点总结、作业事项、疑 难点和复习建议。
5.5 F5 温和提醒与分级反馈
系统采用 三级反馈： 轻度事件 优先提醒学生本 人，例如屏幕提示或 轻微语音提示 ；中度事件 保存关键帧并在课 后报告
中呈现；持续离座、 持续切屏等 高风险事件 才推送家长端。 该策略避免把系统 做成压迫式监控设备。

家长端 展示每节课的课程摘要、 有效学习时长、离座 次数、疑似分心 次数、课程 页面保持率、手机干扰次数和复习建
议。家长端 默认不 显示连续视频，只 显示结构 化摘要和 必要关键帧。

时，灯 带变为蓝色；课程结 束时，屏幕 展示总结 二维码。
6 算法设计与端侧实现
6.1 学习状态评分
第一版不建议 直接使用复杂 深度学习模型判断“认 真程度”，而应采用可解 释的规则评分。系统可以定义学习状态 指
数 LearnScore，用 于反映当前学习过程是 否稳定：
LearnScore = 0.35 * S_seat + 0.25 * S_screen + 0.20 * S_pose + 0.10 * S_write + 0.10 * S_environment
其中 S_seat 表示在座状态，S_screen 表示课程 页面保持情况，S_pose 表示面向屏幕与 趴桌风险，S_write 表示作业 区
域是否有书写/翻页等变化，S_environment 表示灯光和环境 稳定性。分 数低于阈值时 触发本地提醒， 连续低分 才进入家长
报告。

6.3 端侧任务划分
ESP32-P4 端侧建议采用 FreeRTOS 多任务结构。摄像头 任务负责持续采集和低分 辨率预览；音频 任务负责课程 片段
缓存和语音提示 ；传感器 任务周期读取 PIR、光 照、压力等 数据；事件 任务根据 规则生成结构 化事件 ；网络任务负责 
MQTT/HTTPS 上报 ；UI 任务负责屏幕 刷新；执行器 任务负责蜂鸣器、LED 和继电器。网 络请求不能 阻塞本地提醒， 即使
断网，系统也应能 继续记录事件并 进行本地反馈。

// ESP32-P4 端侧事件结构示例：检测到持续离座后生成事件并上报
void check_leave_event() {
    bool pir_empty = read_pir_state() == 0;
    bool pressure_empty = read_seat_pressure() < PRESSURE_TH;
    bool vision_empty = person_detected_recently == false;
    if (pir_empty && pressure_empty && vision_empty) {
        leave_counter++;
    } else {
        leave_counter = 0;
    }
    if (leave_counter > LEAVE_THRESHOLD_SEC) {
        Event e;
        e.type = "leave_seat";
        e.risk = "middle";
        e.timestamp = get_time_ms();
        e.frame_path = save_key_frame();
        mqtt_publish("guardian/event", event_to_json(e));
        speak("检测到离座，请尽快回到课程 ");
    }
}
7 云平台与课程总结智能体
7.1 平台组成
云平台 由 MQTT 消息服务、事件 数据库、对 象存储、课程总结智能体和 前端看板组成。ESP32-P4 将学习事件、关键
帧路径、课程音频 片段和传感器摘要上传到平台 ；平台 调用 ASR 将课程音频 转成文字，再调用 LLM 生成课程总结 ；家长
端读取结构 化报告。
MQTT Topic 用途 主要字段
guardian/{device_id}/telemetry 实时状态 在线、课程模 式、在座状态、 当前风险等 级
guardian/{device_id}/event 事件上报 离座、切屏、 手机干扰、趴桌、课程结 束
guardian/{device_id}/summary/request 总结 请求 课程 开始/结束时间、音频 路径、关键帧 列表
guardian/{device_id}/summary/result 总结结 果 知识点、作业、注 意事项、家长建议
guardian/{device_id}/control 控制指令提醒 开关、隐私模 式、继电器控 制、设备重
启
7.2 课程总结智能体输出格式
课程总结智能体不应只生成一 段长文本，而应 输出结构 化结果，方 便家长和学生 快速阅读。建议 包含：课程 主题、本
节知识点、 老师强调内容、作业事项、学生学习状态摘要、疑似分心 片段、课 后复习建议和 下一次学习提醒。
报告字段 说明
课程 主题 通过 ASR 文本和屏幕关键帧识别课程 标题或主要章节
知识点总结 用 3-6 条短句总结本 节课主要内容
作业提醒 提取“作业、提 交、练习、 背诵、预习”等关键 词附近内容
学习状态 汇总有效学习时长、离座 次数、分心 次数、课程 页面保持率
家长建议 用温和语 气给出复习和习 惯建议，不 使用惩罚性表达

9 可行性、风险与伦理边界
9.1 工程可行性
本方案不要求 ESP32-P4 本地 运行完 整大模型， 因此工程 难度可控。 第一版只需要完成摄像头采集、关键帧 保存、传
感器 读取、事件 规则判断、蜂鸣器/屏幕反馈、MQTT/HTTPS 上传和云端总结 调用， 即可形成完 整 Demo。视 觉识别中的
复杂语义判断可以 由云端 VLM 完成，本地 先用规则和轻量检测保证基础效果。
9.2 技术风险
风险 表现 缓解策略
网络不稳定 云端 VLM/ASR 调用失败或延迟高本地 保留基础提醒，云端 失败时延迟生成
报告
摄像头角 度受限 看不到屏幕或桌面关键 区域固定支架，提 供安装校准界面
误判分心 学生 短暂低头或 思考被误判采用 持续时间 阈值和多传感器 交叉验证
大模型 回答不稳定 总结 遗漏或描述不 准确限定输出模板，保留音频 转写和关键帧 证
据
隐私 争议 家长过 度查看连续画面默认不 直播，报告 优先，关键帧 脱敏与定
期删除

智学 Guardian 方案讨论稿9.3 隐私与伦理原则
本项目 必须明确隐私边界： 默认不 进行连续远程直播；默认不 保存长期完整视频 ；只在事件 触发或课程总结需要时 保
存关键帧 ；家长端 优先展示摘要报告 ；学生应能 看到系统 当前处于何种模式；数据应设定自 动过期删除策略；系统只 给出
行为线索，不 给学生 贴“懒惰”“不认 真”等 标签。

11 总结
“智学 Guardian”将通用多模态 AIoT 终端收窄到家庭网课和学生作业这一具体场景， 避免了原方案过 于宏观的 问
题。系统 围绕“专注守护、温和提醒、课程总结、家长反馈”建立完 整闭环， 既能体现 ESP32-P4 的摄像头、音频、屏幕
和物联网控 制能力，也能体现云端 大模型在课程理解和报告生成中的价值。
作为比赛 选题，本方案的 优势在于场景 清晰、硬件可落地、演示 链路完整、AI 与 IoT 结合 紧密。后续小组讨论时，
可以重点 确定三个问题：是 否选择该方案作为 主选题；是否加入座 椅压力传感作为在座 辅助；课程总结部分 使用哪种云端 
ASR/VLM/LLM 接口。
